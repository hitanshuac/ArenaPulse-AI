# Council Advisors

## Framed Question
The current `agentic-refactor.md` workflow focuses primarily on Architecture Decomposition (God Objects, Separation of Concerns). However, it has been ineffective at systematically cleaning up 'nonsense code' (dead files, orphaned assets, unused dependencies, stale CI workflows). In a recent successful cleanup, the agent instead used a 'deterministic hierarchical linear scan': it performed broad-to-narrow directory recon, read config files, and crucially, ran extensive regex and exact-match `grep_search` operations to mathematically prove dependencies and files were unused *before* drafting a deletion plan. It then halted for explicit human approval before executing. 
Should we update the `agentic-refactor.md` workflow to incorporate this exact deterministic scan methodology for dead-code removal, or should we create a separate workflow for it? What is the right architectural move to ensure 'nonsense code' is effectively caught and deleted without bloating the existing refactor workflow?

## The Contrarian
Updating `agentic-refactor.md` to include a full hierarchical linear scan for dead code is a terrible idea. You are combining two entirely different risk profiles and cognitive tasks. Architectural decomposition is a high-context, subjective process of untangling active logic (God Objects, state management). Dead code removal is a low-context, highly objective, deterministic mathematical proof of absence. If you jam the deterministic scan into the refactoring workflow, the agent will get distracted by orphaned assets and stale CI pipelines while it is supposed to be focusing on decoupling the data flow. The refactor workflow is already 5 dense phases. Adding a massive `grep_search` reconnaissance loop to Phase 1 will blow up the context window and cause the LLM to hallucinate or skip steps. You need a dedicated, isolated workflow for this.

## The First Principles Thinker
What are we actually trying to solve? The user wants "nonsense code" removed reliably. The current workflow fails because its core premise is structural reorganization, not garbage collection. Garbage collection requires rigorous, exhaustive dependency mapping (the "hierarchical scan" and regex `grep`). Reorganization requires conceptual mapping (Domain vs. State vs. Presentation). These are fundamentally different operations. If you merge them, you dilute both. The first principle of modularity applies to workflows too: do one thing well. The right architectural move is to recognize that "Refactor" (restructuring active code) and "Prune" (deleting dead code) are distinct lifecycle events. They should be distinct workflows, perhaps orchestrated by a master sync, but never merged. 

## The Expansionist
Why separate them when they amplify each other? Dead code is the ultimate "Phantom Architecture." You can't effectively decompose a God Object if 30% of its imports are dead and 4 of its helper files are orphaned. The deterministic scan is exactly what Phase 1 of `agentic-refactor.md` has been missing! By updating the existing workflow to *start* with a deterministic linear scan and `grep`-based garbage collection, you shrink the codebase *before* the LLM attempts the complex architectural untangling. It makes the subsequent decomposition phase vastly more accurate because the context window is no longer polluted by dead code. We should aggressively expand Phase 1 of `agentic-refactor.md` into a "Sanitization & Scan" phase. It will make the entire refactoring process 10x more powerful.

## The Outsider
As someone looking at this cold, "Agentic Refactor" sounds like changing how the code works, not deleting old image files and GitHub workflows. If I ask an agent to "refactor this," I expect it to clean up my functions, not delete my stale `.jpg` files and prune `requirements.txt`. The recent successful cleanup you described sounds like a "Codebase Audit" or "Garbage Collection" task. The fact that `agentic-refactor.md` currently mentions "Phantom Architecture" is probably what's causing the confusion. Just take "Phantom Architecture" out of the refactor workflow completely, put it in a new workflow called `dead-code-cleanup.md`, and make the instructions incredibly explicit about using `grep_search` before deleting.
