# ADR-0004: Strict Code Quality & SRE Enforcements

## Context
During the Hack2Skill compliance phase (Phase 1 & 2 of `master-sync`), the codebase failed Radon Cyclomatic Complexity (CC) tests and Pylint standards. Pylint flagged broad exception handlers (`except Exception:`) across the LLM service module, violating Defensive Programming rules, while Radon flagged several functions exceeding `B` (complexity score > 5).

## Decision
1. **Helper Extractions for CC:** We enforce strict separation of concerns by extracting inline JSON parsing, regex error recovery, and DuckDB initialization logic into private helper functions (`_parse_groq_result`, `_handle_groq_error`, `_fetch_history`).
2. **Controlled Broad Exceptions:** While Defensive Programming forbids silent catch-alls, the unpredictable nature of external API SDKs (Groq) and `eval()` math interception requires generic fallback layers. We have explicitly disabled pylint checks for these specific blocks (`# pylint: disable=broad-exception-caught`) *only* when the exception is guaranteed to be piped into the `log_error` observability layer prior to returning a safe default.
3. **Flake8 Conformance:** Inline pylint disables must adhere to strict PEP8 formatting (two spaces preceding the comment).

## Status
Accepted

## Consequences
- The orchestrator (`app.py`) achieved an 'A' grade in complexity.
- LLM parser and Groq extractors are constrained to B(<=8), passing the `test-automation` gate.
- No silent failures remain in the application layer. All edge cases emit telemetry to `error_logs.json`.
